# ADR-0003: Поддерживать серверные ИБ для всех инструментов

- Статус: `accepted`
- Дата: `2026-04-20`

## Контекст

`v8-runner` автоматизирует локальные сценарии 1С через CLI и MCP, но реальные проекты могут использовать не только файловые, но и серверные информационные базы.
Часть текущей реализации и документации пока ориентирована на файловые ИБ: например, `init` создает файловую ИБ, а `IBCMD` сейчас валидируется как file-only backend.

Если оставить server infobase support как частную возможность отдельных команд, появятся разные контракты для `build`, `test`, `dump`, `syntax`, `launch`, `load`, `artifacts`, `init`, `extensions` и MCP tools.
Это противоречит целевому принципу взаимозаменяемости builder backend из ADR-0001 и усложнит автоматизацию через агентов.

## Решение

Принять поддержку серверных информационных баз как обязательный целевой контракт для всех инструментов `v8-runner`.

Правила:

1. Все публичные CLI-команды и MCP tools должны проектироваться так, чтобы работать с серверной ИБ, если соответствующая операция платформы 1С принципиально поддерживает серверное подключение.
2. Файловая ИБ не должна быть единственным архитектурным happy path.
3. Ограничение "только файловая ИБ" допустимо только как явно описанный временный gap.
4. Новые use case, platform adapters и builder backend должны принимать `connection` как общий контракт подключения, а не как путь к файловой базе.
5. Текущий `IBCMD` file-only contract из ADR-0001 остается текущим gap, но не считается целевой архитектурной нормой.
6. `init` должен различать file-based creation path и server-based provisioning/prerequisite path.
7. Политика создания server-based ИБ зависит от backend contract; для `builder=IBCMD` она уточняется в [ADR-0019](0019-sozdavat-servernuyu-infobazu-cherez-ibcmd-pri-init-pri-otsutstvii.md).
8. Для `format=EDT` команда `init` должна продолжать создавать/import-ить EDT workspace даже при серверной строке подключения, потому что этот шаг не требует создания файловой ИБ.

## Неграницы (Non-goals)

1. Немедленная реализация server-based support во всех командах в рамках этой фиксации решения.
2. Автоматическое создание серверного кластера, пользователей или прав доступа; server-based ИБ может получать отдельный backend-specific provisioning contract последующим ADR.
3. Хранение паролей или секретов вне существующей конфигурационной модели.
4. Удаленная установка платформенных утилит 1С или управление сервером 1С.

## Последствия

1. Документация должна явно различать "текущую поддержку" и "целевой контракт server infobase support".
2. Любой новый file-only сценарий должен быть обоснован и отмечен как gap.
3. Валидация конфигурации не должна закреплять file-only ограничения шире, чем это реально требуется конкретным backend-ом или командой.
4. Tests и live smoke должны постепенно получать server-connection сценарии для каждого публичного инструмента.
5. Platform adapters должны строить аргументы подключения через общую модель `V8Connection`, а не через ad hoc работу с `File=...`.

## План реализации

При реализации этого ADR следующими изменениями:

1. Составить матрицу текущих file-only gaps по всем CLI-командам и MCP tools.
2. Разделить валидацию подключения на общую валидацию строки подключения и backend-specific ограничения.
3. Для `DESIGNER` и Enterprise-based операций проверить передачу серверной строки подключения через общий `V8Connection`.
4. Для `IBCMD` исследовать и реализовать server-based подключение либо зафиксировать конкретное платформенное ограничение отдельным gap.
5. Для `init` отделить сценарий создания файловой ИБ от server-based provisioning/prerequisite path и не превращать server connection в hardcoded failed flow.
6. Добавить regression/live tests для server connection там, где это возможно в локальном окружении.

## Верификация

- [x] Целевой контракт server infobase support зафиксирован как архитектурное требование.
- [x] Текущие file-only ограничения описаны как gaps, а не как постоянная архитектурная норма.
- [x] `IBCMD` file-only ограничение из ADR-0001 связано с целевым требованием этого ADR.
- [x] Зафиксировано, что server-based provisioning policy может уточняться последующим backend-specific ADR, включая ADR-0019 для `builder=IBCMD`.
- [x] Зафиксировано, что `init` должен продолжать EDT workspace step и для server connection.
