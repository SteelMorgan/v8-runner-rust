# ADR-0004: Автообнаруживать компоненты платформы 1С по версии-маске

- Статус: `accepted`
- Дата: `2026-04-20`

## Контекст

`v8-runner` вызывает несколько утилит платформы 1С: `1cv8`, `1cv8c`, `ibcmd`, а также отдельный EDT CLI.
Пользовательские окружения часто содержат несколько установленных версий платформы 1С, и жесткое указание полного пути к бинарям делает конфигурацию хрупкой.

Проект уже содержит `platform::locator` и `PlatformUtilities`, которые централизуют поиск утилит.
Нужно зафиксировать целевой контракт: проект сам ищет установленные компоненты платформы 1С на основании версии или версии-маски, а сценарии не должны вручную собирать пути к платформенным бинарям.

## Решение

Принять централизованный автопоиск компонентов платформы 1С как обязательный архитектурный контракт.

Правила:

1. Все use case и platform adapters должны получать пути к `1cv8`, `1cv8c`, `ibcmd` через общий locator/facade, а не через ad hoc поиск.
2. `tools.platform.version` задает требование к версии платформы 1С.
3. Версию не обязательно указывать полностью. Поддерживаемые формы:
   - `8.3.27.1789` — точная версия;
   - `8.3.20` — маска `major.minor.patch`, выбрать максимальную найденную `8.3.20.*`;
   - `8.3` — маска `major.minor`, выбрать максимальную найденную `8.3.*.*`.
4. Если задана версия-маска, locator должен выбрать максимальную доступную установленную версию, удовлетворяющую маске.
5. Если `tools.platform.version` не задана, locator должен выбрать максимальную доступную установленную версию из известных корней поиска.
6. `tools.platform.path` может сужать область поиска: конкретный бинарь, каталог `bin`, корень установки или корень с versioned subdirectories.
7. Для `1cv8`, `1cv8c` и `ibcmd` должен применяться один и тот же принцип выбора версии.
8. Будущий `DESIGNER` agent mode должен использовать этот же механизм discovery, а не отдельную схему поиска.

## Неграницы (Non-goals)

1. Управление установкой платформы 1С или автоматическая загрузка недостающих компонентов.
2. Выбор версии по feature detection вместо версии-маски.
3. Поддержка несовместимых схем каталогов без явного расширения locator.
4. Изменение отдельного EDT discovery contract; EDT CLI может оставаться lenient version-like search, но должен идти через общий facade.

## Последствия

1. Конфигурации проектов могут быть переносимее: вместо полного пути достаточно указать `tools.platform.version`.
2. Тесты locator должны покрывать exact version, `major.minor.patch` mask, `major.minor` mask и выбор максимальной версии.
3. Ошибки поиска должны оставаться явными: если подходящая установленная версия не найдена, команда должна завершиться ошибкой discovery, а не молча брать неподходящую версию.
4. Новые platform adapters не должны принимать путь к бинарю напрямую, если этот путь можно получить через `PlatformUtilities`.

## План реализации

При реализации и сопровождении этого ADR:

1. Расширить `PlatformVersionRequirement`, если текущая реализация не принимает `major.minor`.
2. Проверить выбор максимальной версии для масок `8.3`, `8.3.20`, `8.3.27.1789`.
3. Убедиться, что `1cv8`, `1cv8c` и `ibcmd` проходят через единый locator.
4. Добавить regression tests на поиск по `tools.platform.path` как root/hint и по стандартным roots.
5. Обновлять публичную документацию при изменении supported version mask forms.
