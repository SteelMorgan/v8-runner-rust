# ADR-0001: Границы поддержки IBCMD как ограниченного backend

- Статус: `accepted`
- Дата: `2026-04-02`

## Контекст

В проекте `IBCMD` уже используется как альтернативный backend, но публичный контракт был размазан между кодом и несколькими документами.
Это повышало риск дрейфа формулировок и неоднозначной трактовки поддержки.

Нужно зафиксировать единую границу: какие сценарии поддерживаются сейчас, какие ограничения считаются текущими исключениями, и какой целевой принцип должен удерживаться при развитии builder-слоя.

## Решение

Принять `IBCMD` как **ограниченный сейчас, но архитектурно взаимозаменяемый backend** со следующими границами:

1. Поддерживаемые сценарии: `init`, `build`, `dump`, `extensions`.
2. Подключение к ИБ: только файловая информационная база.
3. `dump --mode partial`:
- object-scoped partial dump нативно не поддержан;
- запрос `partial` деградирует в `incremental export` с предупреждением;
- в результирующем payload сохраняется запрошенный режим `PARTIAL`.
4. Все builder backend должны проектироваться как взаимозаменяемые для реализованных builder-сценариев.
5. Функциональность, реализованная для `DESIGNER` builder, должна быть доступна для `IBCMD` builder либо иметь явное временное исключение в документации/ADR.
6. Будущий `DESIGNER` в режиме агента должен добавляться как еще один backend-способ исполнения без изменения use-case contract.
7. File-only ограничение `IBCMD` является текущим gap и должно развиваться в сторону server-based support согласно ADR-0003.

## Неграницы (Non-goals)

1. Немедленное добавление object-scoped partial dump для `IBCMD`.
2. Немедленная поддержка server-based connection для `IBCMD`.
3. Реализация `DESIGNER` agent mode в рамках этой фиксации решения.

## Последствия

1. Публичная документация должна описывать текущие ограничения `IBCMD`, не превращая их в постоянный отказ от builder-взаимозаменяемости.
2. Внутренние и публичные документы должны обновляться синхронно при изменении границ.
3. Новые builder-сценарии должны проектироваться через общий use-case contract и platform adapter boundary, чтобы `DESIGNER`, `IBCMD` и будущий `DESIGNER` agent mode могли развиваться без расхождения публичного поведения.
4. Если конкретная возможность временно доступна только через один builder, это должно быть явно зафиксировано как gap, а не как новая архитектурная норма.
5. Поддержка серверных ИБ для `IBCMD` должна рассматриваться как часть общего целевого контракта всех инструментов, а не как необязательное расширение.

## План реализации (для этой фиксации решения)

1. Создать ADR-каталог и индекс решений.
2. Зафиксировать ссылку на этот ADR в пользовательских и внутренних обзорных документах.
3. Синхронизировать формулировки в:
- `README.md`
- `docs/CAPABILITIES.md`
- `docs/DEEP_DIVE.md`
- `ARCHITECTURE.md`
