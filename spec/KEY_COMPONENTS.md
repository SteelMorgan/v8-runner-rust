# Ключевые компоненты и их функциональность

Документ собран по актуальному коду проекта. Его задача: зафиксировать состав системы, границы ответственности компонентов и критичные технические детали, которые важны для переписывания на другой стек.

## 1. Общая карта компонентов

Архитектурно система делится на 6 крупных зон:

1. Точка входа и конфигурация приложения.
2. MCP-слой.
3. Оркестрация сценариев.
4. Подсистема анализа изменений и временного состояния.
5. Адаптеры к платформе 1С и EDT.
6. Подсистемы парсинга результатов: тесты, синтаксис, логи.

## 2. Компоненты верхнего уровня

| Компонент | Роль | Что получает | Что отдает | Критичные детали |
| --- | --- | --- | --- | --- |
| `Main.kt` | Старт приложения | CLI args, Spring context | Поднятое приложение | Без web UI, профиль `mcp`, кастомная переконфигурация logback |
| `ExternalConfigLoader` | Загрузка внешнего YAML | `SPRING_CONFIG_IMPORT` | PropertySource в Spring | Загружает внешние YAML до создания бинов |
| `ApplicationProperties` | Единая типизированная конфигурация | YAML / env | Валидированная модель настроек | Сразу валидирует `basePath`, `source-set`, connection, builder |
| `McpServer` | Набор MCP-tool методов | Вызовы от MCP-клиента | DTO-ответы tool-методов | Источник реального списка возможностей |
| `McpConfiguration` | Регистрация tool callbacks | `McpServer` | MCP tool provider | Тонкая Spring-обвязка |

## 3. Оркестрация сценариев

### 3.1. `LauncherService`

Главный прикладной координатор.

Ответственность:

- запуск build-пайплайна;
- запуск тестов с обязательной предварительной сборкой;
- запуск приложений 1С;
- переключение между EDT и DESIGNER сценариями;
- сохранение состояния после успешных шагов.

Ключевые методы:

- `build(fullRebuild: Boolean = false)`
- `runTests(request: TestExecutionRequest)`
- `launch(request: LaunchRequest)`

Критичные детали:

- `runTests()` всегда сначала вызывает `build()`;
- для `EDT` build состоит из двух фаз: `convertSources()` и затем `buildSourceSet()`;
- `fullRebuild=true` не меняет команды платформы, а очищает кэши хешей.

### 3.2. `EdtCliStartService`

Компонент фоновой жизни приложения.

Ответственность:

- автозапуск `1cedtcli` при старте приложения;
- создание и хранение единственного интерактивного executor;
- single-flight инициализация процесса;
- мониторинг и перезапуск EDT CLI при сбое.

Критичные детали:

- работает только если `app.format == EDT`;
- включается только при `app.tools.edt-cli.auto-start == true`;
- использует lazy/single-flight запуск через `Deferred`;
- мониторинг живет отдельно от MCP-tool вызовов;
- это единственный настоящий background-компонент в приложении.

### 3.3. `SourceSetsService`

Компонент связывает бизнес-понятие `source-set` с реальными рабочими директориями и storage.

Ответственность:

- выдавать `SourceSetContext` для EDT-исходников;
- выдавать `SourceSetContext` для Designer-исходников;
- в EDT-режиме строить временный Designer-source-set в `workPath`;
- прикреплять отдельное хранилище хешей к каждому логическому источнику.

Критичные детали:

- в `EDT` одновременно существуют два источника истины:
  - исходный EDT source-set;
  - временный Designer source-set после экспорта;
- для `DESIGNER` существует только один контекст.

## 4. MCP-слой

### 4.1. `McpServer`

Это фасад над прикладными сервисами. Он:

- принимает аргументы tool-метода;
- очищает лог при необходимости;
- делегирует работу в сервисы;
- переводит внутренние результаты в MCP DTO.

Реально опубликованные tool-методы:

- `run_all_tests`
- `run_module_tests`
- `build_project`
- `dump_config`
- `launch_app`
- `check_syntax_edt`
- `check_syntax_designer_config`
- `check_syntax_designer_modules`

Что важно при переписывании:

- список tool-методов должен определяться по коду, а не по `README`;
- компактные и полные ответы для тестов это часть MCP-контракта;
- ошибки не пробрасываются "как есть", а упаковываются в структурированные DTO.

### 4.2. DTO и конвертеры

Ключевые DTO:

- `McpTestResponse`
- `McpBuildResponse`
- `McpDumpResponse`
- `McpLaunchResponse`
- `McpSyntaxCheckResponse`

Ключевые конвертеры:

- `RunTestResult.toResponse(full: Boolean)`
- `SyntaxCheckResult.toResponse(checkName, duration)`

Критичные детали:

- compact/full режим тестового ответа меняет состав `testDetail`;
- compact режим скрывает passed-тесты и урезает stack trace;
- ошибки синтаксической проверки уже возвращаются как `issues`, а не просто сырой stdout.

## 5. Подсистема анализа изменений

### 5.1. `FileSystemChangeAnalysisAction`

Высокоуровневый action для определения, есть ли изменения и к каким `source-set` они относятся.

Что делает:

1. Создает `FileBuildStateManager`.
2. Запускает проверку изменений.
3. Формирует шаги (`ActionStepResult`) для последующего отчета.
4. Группирует изменения через `SourceSetChangeAnalyzer`.
5. После успеха сохраняет новые хеши и timestamp.

Важно:

- анализ вызывается по требованию;
- отдельного persistent watcher-процесса нет.

### 5.2. `FileBuildStateManager`

Низкоуровневый движок change detection.

Алгоритм:

1. Берет timestamp последней успешной обработки `source-set`.
2. Через `Scanner` находит кандидатов по `lastModified`.
3. Для кандидатов сверяет текущий hash с сохраненным hash.
4. Возвращает только реальные изменения.

Критичные детали:

- при ошибке сканирования делает fallback: считает все исходники измененными;
- работает по каждому `SourceSetContext` отдельно;
- возвращает не только путь файла, но и `ChangeType` + новый hash.

### 5.3. `Scanner`

Простой файловый сканер.

Что делает:

- рекурсивно обходит каталоги;
- фильтрует служебные пути;
- по timestamp отбирает потенциально измененные файлы.

Что не делает:

- не слушает файловые события ОС;
- не работает как background watcher;
- не хранит состояние сам по себе.

### 5.4. `SourceSetChangeAnalyzer`

Группирует общий набор изменений по логическим подпроектам из `source-set`.

Зачем нужен:

- build/convert работают не на уровне "весь проект", а на уровне конфигурации и отдельных расширений;
- позволяет конвертировать и загружать только затронутые куски.

### 5.5. `HashStorage`

Фактическое хранилище сейчас `redb`-based.

Роли:

- `HashStorage` хранит hashes/timestamps и служебные метаданные;
- storage изолируется по имени logical source-set (`edt`, `designer`).

Критичные детали:

- путь хранения: `workPath/hash-storages/*.redb`;
- запись происходит на каждом обновлении;
- старые docs могут называть это MapDB/JSON, но код уже переехал на `redb`.

## 6. Подсистема сборки и конвертации

### 6.1. `BuildAction` и его реализации

Абстракция сборки:

- `DesignerBuildAction`
- `IbcmdBuildAction`

Общая логика лежит в `AbstractBuildAction`.

Что делает общий слой:

- измеряет время;
- выбирает partial/full режим для каждого `source-set`;
- последовательно загружает конфигурацию и расширения;
- собирает шаги и ошибки в `BuildResult`.

### `DesignerBuildAction`

Использует `DesignerDsl`.

Особенности:

- отключает стартовые диалоги и сообщения;
- для частичной загрузки генерирует `listFile`;
- при загрузке вызывает `updateConfigDumpInfo()` и `updateDBCfg()`;
- отдельного финального шага `updateDb()` нет, потому что обновление вшито в команды загрузки.

### `IbcmdBuildAction`

Использует `IbcmdDsl`.

Особенности:

- для partial использует `config import files --partial`;
- после загрузки вызывает отдельный `config apply()`;
- завязан на извлечение `File=` из строки подключения.

Критичный риск:

- текущая реализация IBCMD ориентирована на файловую базу.

### 6.2. `EdtInteractiveConvertAction`

Конвертер `EDT -> Designer`.

Что делает:

- для каждого измененного EDT-проекта вызывает `1cedtcli export`;
- выгружает проект в каталог Designer-source-set с именем `sourceSet.name`;
- фиксирует результат по каждому подпроекту.

Критично:

- конвертация не идет "в базу";
- это промежуточный экспорт в локальную рабочую директорию.

### 6.3. `PartialLoadListGenerator`

Один из самых важных технических компонентов build-пайплайна.

Что делает:

- получает набор измененных файлов;
- преобразует их в `listFile` для частичной загрузки;
- для `.bsl` добавляет связанные XML и каталог объекта;
- запрещает partial, если затронут `Configuration.xml`;
- запрещает partial, если изменений слишком много.

Это логика, которую нельзя "заменить на что-то похожее", не изменив поведение системы.

## 7. Подсистема выгрузки из ИБ

### 7.1. `DumpService`

Прикладной фасад над `DumpAction`.

Что делает:

- определяет целевой каталог по `source-set` и `extension`;
- нормализует режим выгрузки;
- валидирует наличие списка объектов для `PARTIAL`.

### 7.2. `DumpAction` и его реализации

Реализации:

- `DesignerDumpAction`
- `IbcmdDumpAction`

### `DesignerDumpAction`

Использует `DesignerDsl`.

Особенности:

- полная выгрузка и "инкрементальная" логика идут через `DumpConfigToFiles`;
- для incremental важна роль `ConfigDumpInfo.xml`;
- для `PARTIAL` создается временный файл со списком объектов.

Замечание:

- различие между `FULL` и `INCREMENTAL` в Designer-backend опирается на семантику платформы и наличие `ConfigDumpInfo.xml`;
- в коде оба режима используют очень близкий набор аргументов.

### `IbcmdDumpAction`

Использует `IbcmdDsl`.

Особенности:

- `FULL` через `--force`;
- `INCREMENTAL` через `--sync`;
- настоящий partial dump по объектам не реализован напрямую.

## 8. Подсистема тестирования YaXUnit

### 8.1. `YaXUnitRunner`

Низкоуровневый исполнитель тестов.

Что делает:

- строит временный JSON-конфиг YaXUnit;
- сохраняет его на диск;
- запускает 1С:Предприятие с `RunUnitTests=<configPath>`;
- возвращает пути к отчету и логам.

### 8.2. `YaXUnitTestAction`

Высокоуровневый action тестирования.

Что делает:

- вызывает `YaXUnitRunner`;
- разбирает JUnit XML через `ReportParser`;
- достает ошибки из YaXUnit-лога через `LogParser`;
- формирует `RunTestResult`.

### 8.3. `YaXUnitConfig`

Слой трансляции доменного запроса в JSON-конфиг запуска.

Поддерживает:

- запуск всех тестов;
- запуск тестов одного модуля;
- внутренне есть и `RunListTestsRequest`, но он не опубликован как MCP-tool.

### 8.4. `ReportParser`

Минимальный JUnit XML parser.

Что возвращает:

- summary;
- suites;
- cases;
- статусы `PASSED/FAILED/SKIPPED/ERROR`.

### 8.5. `LogParser`

Потоковый парсер YaXUnit-лога.

Что делает:

- читает лог построчно;
- выделяет блоки ошибок по маркеру `[ERR]`;
- собирает многострочные error-блоки без загрузки всего файла в память.

## 9. Подсистема синтаксических проверок

### 9.1. `SyntaxCheckService`

Единый фасад над тремя типами проверок:

- EDT validate;
- Designer CheckConfig;
- Designer CheckModules.

Что делает:

- строит команды;
- запускает их через соответствующие DSL;
- пишет/читает лог-файлы;
- преобразует вывод в `SyntaxCheckResult`.

### 9.2. Парсеры ошибок

### `DesignerValidationLogParser`

Разбирает два типа записей:

- ошибки модулей с координатами;
- ошибки объектов без координат.

### `EdtValidationLogParser`

Разбирает EDT-лог как табличный формат и превращает его в `Issue.EdtValidationIssue`.

Критично:

- value для MCP-клиента это именно структурированные `issues`, а не raw output.

## 10. Адаптеры к платформе 1С

### 10.1. `PlatformDsl`

Фабрика специализированных DSL:

- `designer()`
- `ibcmd()`
- `enterprise()`
- `edt()`

Это центральная точка, через которую прикладной слой получает доступ к платформе.

### 10.2. `DesignerDsl`, `IbcmdDsl`, `EnterpriseDsl`, `EdtDsl`

Назначение:

- `DesignerDsl` — команды Конфигуратора;
- `IbcmdDsl` — команды `ibcmd`;
- `EnterpriseDsl` — запуск клиентских приложений и YaXUnit;
- `EdtDsl` — команды `1cedtcli`.

Общее свойство:

- они не содержат бизнес-логики сценария;
- они строят команды и сразу их выполняют.

### 10.3. `ProcessExecutor`

Универсальный исполнитель внешних процессов.

Что делает:

- запускает процесс;
- читает `stdout`/`stderr`;
- опционально пишет и потом читает лог-файл 1С;
- умеет запускать асинхронно для `launch_app`;
- возвращает `ProcessResult`.

### 10.4. `InteractiveProcessExecutor`

Специализированный интерактивный исполнитель для `1cedtcli`.

Что делает:

- ждет prompt `1C:EDT>`;
- держит один живой процесс;
- посылает команды в stdin;
- читает ответ до следующего prompt;
- умеет корректно завершать или форсированно убивать процесс.

Это важнейшая техническая оптимизация EDT-пайплайна.

### 10.5. `EdtCliExecutor`

Адаптер между `InteractiveProcessExecutor` и интерфейсом `CommandExecutor`.

Нужен для того, чтобы `EdtDsl` мог работать так же, как и обычные DSL, но через интерактивный backend.

## 11. Поиск бинарников платформы

### 11.1. `PlatformUtilities`

Сервис-обертка над поиском утилит и выбором executor.

Что делает:

- подставляет фактическую версию платформы или EDT;
- ищет binary нужного utility type;
- для EDT при `auto-start=true` выдает интерактивный executor;
- для остальных случаев выдает обычный `ProcessExecutor`.

### 11.2. `UtilityLocator` и `SearchStrategy`

Подсистема поиска исполняемых файлов на Windows/Linux/macOS.

Особенности:

- кэширует найденные пути;
- выбирает лучший кандидат по версии;
- для платформенных утилит требует точную версию `major.minor.patch.build`;
- для EDT может использовать `latest`.

Критично:

- логика поиска платформы это отдельная ответственность, ее лучше не смешивать с business-layer при переписывании.

## 12. Логирование и временные данные

### 12.1. `LogManager`

Функция узкая, но важная:

- по env-флагу `CLEAN_BEFORE_EXECUTION=true` очищает лог-файлы перед каждым tool-вызовом.

### 12.2. Временные артефакты

Система создает:

- временные log-файлы 1С;
- временные `listFile` для partial load/dump;
- временный JSON-конфиг YaXUnit;
- рабочую директорию `workPath` для EDT;
- `redb`-хранилища хешей.

При переписывании важно сохранить управляемую и предсказуемую схему временных файлов.

## 13. Критичные технические детали и ограничения

1. Источник истины это код, а не `README.md` и не `docs/FEATURES.md`.
2. Фонового file watcher нет; есть только on-demand scan.
3. Для EDT критична интерактивная долгоживущая CLI-сессия.
4. В EDT-режиме есть отдельная временная Designer-копия проекта.
5. Хранилище изменений уже `redb`, хотя в старых документах еще встречается `MapDb`/JSON.
6. IBCMD-backend в build-пайплайне зависит от файловой ИБ.
7. Designer и IBCMD дают разную семантику dump/build, это не просто две interchangeable реализации.
8. Тестовый пайплайн завязан на временный YaXUnit JSON-конфиг и JUnit XML parsing.
9. MCP-ответы уже содержат структурированные данные; переписывание не должно деградировать до "строка stdout".

## 14. Рекомендуемая декомпозиция для переписывания

Если переносить проект на другой стек, разумно сохранить те же границы:

1. `Transport/MCP Layer`
2. `Use Cases / Orchestration Layer`
3. `Change Detection Layer`
4. `1C Platform Adapters`
5. `EDT Session Manager`
6. `Result Parsers`
7. `Persistent Build State Storage`

Это позволит:

- не смешать MCP-протокол с логикой сборки;
- отдельно тестировать пайплайны build/test/dump;
- заменить platform adapters без переделки бизнес-логики;
- сохранить текущую наблюдаемую семантику инструмента.
